## 3.2 技术栈

| 层级 | 技术 | 版本 | 说明 |
|------|------|------|------|
| 前端框架 | Flutter | 3.x | Dart语言，跨平台UI |
| 前端状态管理 | Riverpod | 2.x | 依赖注入 + 状态管理 |
| 前端本地DB | sqflite | 2.x | Flutter SQLite插件 |
| 前端HTTP | dio | 5.x | HTTP客户端 |
| 后端框架 | FastAPI | 0.110+ | Python异步Web框架 |
| 后端ORM | SQLAlchemy | 2.x | 异步ORM |
| 后端DB | MySQL | 8.0 | 关系型数据库 |
| 后端DB驱动 | aiomysql | - | MySQL异步驱动 |
| LLM SDK | openai / litellm | - | 可配置多种LLM Provider |
| 导出 | reportlab | - | PDF生成（营养报告导出用） |

## 3.3 数据分布设计

### 策略：本地为主，服务端为辅

```
┌────────────────────────────────────────────────────┐
│              数据存储分布                           │
├──────────────┬─────────────────────────────────────┤
│   Flutter    │              后端                    │
│   (SQLite)   │           (MySQL)                   │
├──────────────┼─────────────────────────────────────┤
│ 每周菜单 ✓    │ 菜品模板 (含食材配置 + 营养数据) ✓    │
│ 本地食物库 ✓  │ LLM生成结果缓存 ✓                    │
│ 自定义分类 ✓  │ 系统配置 ✓                           │
│ 用户设置 ✓    │                                      │
├──────────────┴─────────────────────────────────────┤
│ 菜品创建流程：                                       │
│  1. 用户在Flutter输入菜品名                         │
│  2. Flutter发送POST /api/dishes/generate 到后端     │
│  3. 后端调用LLM生成食材配置+营养数据，存入MySQL       │
│  4. 后端返回生成的菜品数据给Flutter                  │
│  5. Flutter将菜品保存到本地SQLite                   │
│                                                     │
│ 采购清单流程：                                       │
│  1. 用户在Flutter选择菜单触发                        │
│  2. Flutter将菜单菜品数据发送到后端                   │
│  3. 后端计算合并食材 + 自动分类，返回分组数据         │
│  4. Flutter按分类展示Todo清单，支持勾选               │
│                                                     │
│ 营养分析流程：                                       │
│  1. 用户在Flutter选择菜单触发                        │
│  2. Flutter将菜单营养数据发送到后端                   │
│  3. 后端计算 + 调用LLM评估                          │
│  4. 结果展示在Flutter，可导出PDF                     │
└─────────────────────────────────────────────────────┘
```

## 3.4 LLM收口设计

**原则：LLM能力由服务端统一管理，前端不感知LLM配置。**

- 后端在启动时从 `config.yaml` 加载LLM配置
- LLM Service 提供统一调用接口，屏蔽不同Provider差异
- 支持的Provider（通过配置切换）：
  - OpenAI 兼容API（GPT-4o / GPT-3.5 / DeepSeek 等）
  - Ollama 本地模型
  - 自定义 OpenAI-compatible endpoint
- LLM调用场景：
  - 菜品食材配置生成
  - 菜品营养数据生成
  - 营养评估报告生成
- 前端通过标准API调用，无需知道底层使用哪个LLM
